# Заметки по установке окружения

## BepInEx (Task 2)
- Версия: **BepInEx 5.4.23.5** (win x64, Mono loader)
- Источник: https://github.com/BepInEx/BepInEx/releases/download/v5.4.23.5/BepInEx_win_x64_5.4.23.5.zip
- Smoke-тест: игра запускается, лог `BepInEx/LogOutput.log` показывает `Chainloader startup complete`, ошибок нет.
- Из лога: `Running under Unity v2021.1.16`, `CLR runtime version: 4.0.30319`, `System platform: Bits64, Windows`.
- **`Supports SRE: False`** — System.Reflection.Emit недоступен. Harmony в BepInEx 5 Mono патчит через MonoMod.RuntimeDetour (нативные детуры), не требующие SRE, поэтому патчинг TMP в Task 7/9/10 должен работать. Проверить на spike (Task 7).
- `.gitignore` исключает `/BepInEx/`, поэтому сам фреймворк не версионируется (только поставляемые артефакты перевода будут добавлены позже точечно).
- Включена консоль BepInEx (`BepInEx/config/BepInEx.cfg` → `[Logging.Console] Enabled = true`) — при запуске игры видно живое окно лога.

## XUnity.AutoTranslator (Task 3)
- Версия: **XUnity.AutoTranslator-BepInEx-5.6.1** (Mono).
- Конфиг `BepInEx/config/AutoTranslatorConfig.ini`: `Endpoint=` (пусто — без онлайн/машинного перевода), `Language=ru`, `FromLanguage=en`, `UseStaticTranslations=True`.
- Smoke (из лога): `Loaded XUnity.AutoTranslator into Unity [2021.1.23f1]`, `Version of TextMesh Pro: 1.4.0`, перехват TMP активен.
- **Warning:** хук `TMP_Text_SetCharArray_Hook3` не встал (нет метода `SetCharArray(int[],int,int)` в этой версии TMP). Остальные TMP-хуки встали; основной путь установки текста перехватывается. Если отдельные места не будут переводиться — вернуться к этому.
- **Поведение дампа:** при пустом `Endpoint` AT НЕ авто-дампит непереведённые строки (pipeline пассивен). Поэтому список строк для перевода собираем:
  1. статически — `tools/dump_strings.py` из ассетов (основной источник, Task 12);
  2. при необходимости динамически — локальный echo-endpoint (`CustomTranslate` → localhost), который возвращает строку как есть, давая точный рантайм-список без машинного перевода.
- **Подмена** английского текста русским работает офлайн: AT при старте грузит `Translation/ru/Text/_AutoGeneratedTranslations.txt` (static translations) и подменяет совпадения без обращения к endpoint.

## SPIKE рантайм-загрузки шрифтов (Task 7) — РАБОТАЕТ
- **Тупиковые пути** (проверено в плеере): `Font.CreateDynamicFontFromOSFont` + `TMP_FontAsset.CreateFontAsset(Font)` → NRE даже для системного Arial («Unable to load font face… Include Font Data»); Win32 `AddFontResourceEx(FR_PRIVATE)` регистрирует шрифт в GDI, но Unity его не видит.
- **Рабочий путь**: `FontEngine.LoadFontFace(byte[] ttf, 90)` + ручная сборка `TMP_FontAsset` (`ScriptableObject.CreateInstance`, faceInfo/atlasTextures через публичные сеттеры, остальное — приватные поля `m_*` через reflection, материал на шейдере `TextMeshPro/Distance Field`).
- **Корень бага «0 глифов»**: `TryAddCharacterInternal` на каждый символ заново вызывает `FontEngine.LoadFontFace(m_SourceFontFile, pointSize)`. У рантайм-ассета `m_SourceFontFile == null` → тихий fail, символ попадает в `m_MissingUnicodesFromFontFile`. В этой версии TMP у `TMP_FontAsset` НЕТ приватного метода `LoadFontFace` — вызов FontEngine прямой, патчить нужно сам FontEngine.
- **Фикс**: в `m_SourceFontFile` кладём маркерный `new Font(name)`, регистрируем «маркер → байты ttf» в словаре, Harmony-префиксы на все перегрузки `FontEngine.LoadFontFace(Font, …)` перенаправляют загрузку на байты. Это чинит и пре-бейк, и динамическое добавление глифов на лету.
- **Атлас**: свежая `Texture2D` содержит мусор — очищаем `SetPixelData(new byte[w*h])` перед использованием.
- **Результат**: `pre-baked 67 cyrillic glyphs`, `chars=67 glyphs=67 hasА=True hasё=True` (Ё/ё, А-я, №). Подход для Task 9 (FontService) подтверждён.
