# Адаптивный русификатор Gunsaw — дизайн

- **Дата:** 2026-06-10
- **Игра:** Gunsaw (демо, Windows), разработчик Orsoniks
- **Движок:** Unity 2021.1.23f1, Mono-бэкенд, рендер текста через TextMeshPro
- **Статус:** дизайн утверждён, готов к написанию плана реализации

## 1. Цель и контекст

Сделать **адаптивный русификатор** для Gunsaw. «Адаптивный» в формулировке заказчика означает:

1. **Правильный, качественный перевод** (не машинный «на лету», а выверенный).
2. **Адаптивная вёрстка** — русский текст длиннее английского и не должен обрезаться/вылезать за кнопки и плашки.
3. **Грамотная работа со шрифтами** — добавить кириллицу с **максимально точным сохранением стиля** оригинальных шрифтов. Это критично: «суть в шрифте», особенно в **репликах врагов** и атмосферных надписях.

### Установленная фактура (по результатам разведки)

- Игра — Unity **Mono** (не IL2CPP) → `Assembly-CSharp.dll` декомпилируется и патчится, доступен стабильный BepInEx 5.
- **Встроенной системы локализации нет** (ни I2 Localization, ни Unity Localization, ни словарей языков). Видимый игроку текст лежит в основном в Unity-ассетах (TextMeshPro-компоненты на сценах/префабах) и частично в коде.
- Используется множество кастомных стилизованных TMP SDF-шрифтов: `Gelatik-Regular`, `chloe font v1`, `Acidic`, `conthrax-sb`, `hemi head bd it`, `respawn-pro`, `PixelOperator` / `PixelOperatorOutline`, `Retro Gaming`, `Victor`, `Sansation_Regular`, плюс дефолтный `LiberationSans`. У большинства в SDF-атласе **только латиница** — кириллицу нужно добавлять.
- TMP в Unity 2021.1 поддерживает `TMP_FontAsset.CreateFontAsset(Font)` → кириллические шрифты можно генерировать **в рантайме из `.ttf`**, без Unity Editor и без AssetBundle.

## 2. Зафиксированные решения

| Вопрос | Решение |
|---|---|
| Что значит «адаптивный» | Правильный перевод + адаптивная вёрстка + кириллица с сохранением стиля |
| Источник перевода | Совместно: Claude делает черновик и собирает глоссарий, заказчик выверяет |
| Способ доставки | Рантайм-мод на BepInEx (оригинальные файлы игры не меняются) |
| Архитектура | **Подход C (гибрид)**: XUnity.AutoTranslator как движок перехвата/словаря + тонкий свой плагин для шрифтов и вёрстки |
| Стиль шрифтов | Максимально точное сохранение, по каждому шрифту — триаж (см. §4) |
| Онлайн-перевод | Выключен полностью; весь перевод — только через ручной словарь |
| Глоссарий | Собираем с нуля; Claude предлагает варианты, заказчик утверждает спорные |

## 3. Архитектура и поток данных

### Компоненты (структура папок в корне игры)

```
Gunsaw/
├── BepInEx/                          ← BepInEx 5 (Mono loader)
│   ├── plugins/
│   │   ├── XUnity.AutoTranslator/    ← движок: перехват + словарь (онлайн ВЫКЛ)
│   │   └── GunsawRusFonts.dll        ← наш тонкий плагин (шрифты + вёрстка)
│   ├── config/
│   │   └── AutoTranslatorConfig.ini  ← настройки AT (язык ru, режим offline)
│   └── Translation/ru/Text/
│       └── _AutoGeneratedTranslations.txt  ← словарь en→ru (наш перевод)
└── fonts/                            ← кириллические .ttf + карты
    ├── *.ttf
    ├── font_map.json                 ← "имя игрового шрифта" → "кир. .ttf"
    └── layout_rules.json             ← точечные правила вёрстки
```

### Разделение ответственности

- **XUnity.AutoTranslator** отвечает за «какой текст»: перехват вывода (TMP/UGUI/IMGUI), дамп непереведённого, подмену по словарю. Не дорабатывается — только конфигурируется.
- **Наш плагин `GunsawRusFonts`** отвечает за «как выглядит»: подключение стилевых кириллических fallback-шрифтов к нужным FontAssets и применение правил адаптивной вёрстки.

Части не знают друг о друге и общаются только через результат: AT уже подменил текст → наш плагин чинит шрифт/вёрстку. Каждую часть можно тестировать и отключать отдельно.

### Поток данных для одной строки

```
Игра выставляет текст в TMP_Text.text  ("Press E to interact")
        │
        ▼
XUnity.AutoTranslator перехватывает (Harmony hook)
        ├─ строка есть в словаре? ──ДА──► берёт перевод "Нажмите E…"
        │                          ──НЕТ─► дампит в _AutoGeneratedTranslations.txt,
        │                                  оставляет оригинал (переведём позже)
        ▼
Текст уходит в рендер TMP
        ▼
Наш плагин (пост-обработка компонента):
        ├─ смотрит fontAsset.name ("chloe font v1 SDF")
        ├─ если кириллицы нет → подключает стилевой кир. fallback к этому FontAsset
        └─ применяет правила адаптивной вёрстки
        ▼
На экране: "Нажмите E…" нужным стилевым шрифтом, без обрезки
```

## 4. Подсистема шрифтов (ядро задачи)

### Принцип: fallback, а не замена

Оригинальные FontAssets не трогаем. Для каждого игрового шрифта создаём кириллический TMP-шрифт и **добавляем его в `fontAsset.fallbackFontAssetTable`**. Латиница рисуется оригиналом, кириллица берётся из fallback → стиль латиницы сохраняется, кириллица подбирается под него.

### Метод загрузки `.ttf` в рантайме (технический риск — решается spike-ом на старте)

TMP **не использует `.ttf` напрямую** — ему нужен `TMP_FontAsset` с SDF-атласом. Два рабочих пути, выбираем spike-ом в начале реализации:

1. **Dynamic OS-font (предпочтительно, без Unity Editor):** плагин при старте регистрирует `.ttf` в процессе через Win32 `AddFontResourceEx` → получает `Font` через `Font.CreateDynamicFontFromOSFont(faceName, size)` → создаёт `TMP_FontAsset.CreateFontAsset(font)` (динамический атлас) → подключает как fallback. Не требует Editor.
2. **AssetBundle (запасной, надёжный):** TMP FontAssets собираются в Unity Editor **той же версии 2021.1.23f1**, пакуются в AssetBundle, плагин грузит из него (механизм, который использует сам XUnity.AutoTranslator через `FallbackFontTextMeshPro`). Требует Editor, но гарантированно работает.

Spike (ранняя задача плана) проверяет путь 1 на одном кириллическом шрифте в реальной игре; если качество/стабильность не устраивают — переходим на путь 2.

### Триаж на каждый шрифт (по порядку, останавливаемся на первой успешной ступени)

1. **Идентификация** — определить шрифт по имени, найти происхождение; проверить, не встроён ли исходный `.ttf` прямо в игру (обычно в Unity-билде остаётся только SDF-атлас — проверить надо).
2. **Кириллица в оригинале?** — если в исходном шрифте кириллица уже есть (вероятно у `LiberationSans`, `PixelOperator`), берём её — идеальное совпадение.
3. **Готовая кириллическая версия?** — у многих шрифтов есть официальные/расширенные релизы с кириллицей (например, у `conthrax`). Ищем готовое прежде, чем делать руками.
4. **Делаем сами:**
   - **(основной путь)** подобрать кириллический шрифт в максимально близком стиле;
   - **(для уникальных ключевых, напр. реплики врагов)** дорисовать кириллические глифы на базе латинских форм оригинала в FontForge/Glyphs — часть букв (А В Е К М Н О Р С Т Х а е о р с х) переносится напрямую, остальные конструируются. Только там, где аналога нет, а стиль критичен.

### Утверждение результатов

Результат триажа — таблица «шрифт → решение → источник кириллицы», согласуется до сборки. По ключевым шрифтам (реплики врагов, заголовки) показываются варианты на сравнение; по второстепенным — подбор и показ результата.

### Цепочка fallback (порядок поиска глифа)

`оригинал → стилевой кириллический fallback → нейтральный кириллический fallback → видимый плейсхолдер`

Нейтральный кириллический шрифт гарантирует, что текст никогда не превращается в `□□□`.

## 5. Адаптивная вёрстка

Русский на 15–30% длиннее английского. Плагин чинит вёрстку per-компонент после подмены текста:

- **Авто-уменьшение шрифта в фиксированных рамках** — для кнопок/плашек с жёстким размером включаем `enableAutoSizing` с разумным `fontSizeMin`.
- **Грамотный перенос/overflow** — для многострочных областей (диалоги, описания) включаем word wrapping и подходящий `overflowMode`.
- **Точечные правила-исключения** — `layout_rules.json` задаёт индивидуальные правила по имени объекта/шрифту: множитель размера, отступы, максимум строк.
- **Сохранение атмосферных мест** — где размер/позиция несут стиль (крупный заголовок, реплика врага), правки минимальны: сначала ужать в пределах ±1 кегля, затем расширять перенос.

Принцип эскалации: **самое мягкое вмешательство сначала** (перенос → лёгкий авто-сайз → правило-исключение). Все правки в рантайме. Большинство мест чинятся автоматически; руками — только конкретные проблемные элементы через конфиг.

## 6. Процесс перевода и словарь

### Формат

XUnity.AutoTranslator хранит перевод построчно в `_AutoGeneratedTranslations.txt`:

```
Press E to interact=Нажмите E, чтобы взаимодействовать
Deathmatch=Бой насмерть
```

Правка на лету (AT перечитывает файл), без сборки.

### Этапы наполнения

1. **Дамп всех строк** — проход по игре с AT собирает весь показанный текст; плюс статическое извлечение строк из ассетов/DLL для того, что не успело показаться. Даёт полный список и реальный объём.
2. **Глоссарий терминов** — до перевода фиксируем единый перевод повторяющихся терминов (оружие, режимы, имена, кнопки, UI-команды). Собираем с нуля, спорные утверждает заказчик.
3. **Черновой перевод по экранам** — пакетами по смысловым зонам (меню → настройки → HUD → реплики врагов → туториал), с учётом тона. Реплики врагов — отдельное внимание к стилю.
4. **Выверка заказчиком** — правки формулировок/терминов в файле или списком, итерируем.

### Краевые случаи перевода

- **Подстановки** (`Score: {0}`, `{player} wins`) — плейсхолдеры/числа не трогаем, переводим текст вокруг (AT-правила splitter/regex).
- **Теги форматирования** (`<b>`, `<color>`, спрайты) — разметку не ломаем, переводим только видимый текст.
- **Дубли и регистр** — одинаковые строки переводим один раз; КАПС заголовков в русском применяем осознанно.

## 7. Поставка, обработка ошибок, тестирование

### Поставка

Одна папка, распаковывается в корень игры: BepInEx 5 (x64, Mono) + XUnity.AutoTranslator + `GunsawRusFonts.dll` + сконфигурированный `AutoTranslatorConfig.ini` (ru, онлайн ВЫКЛ) + словарь + `fonts/` (`.ttf` + `font_map.json` + `layout_rules.json`) + краткий `README`.

### Обработка ошибок и устойчивость

- **Шрифт не найден/не загрузился** → лог + откат на нейтральный кириллический fallback, текст читаем, игра не падает.
- **Нет перевода строки** → показывается оригинал, строка дампится для перевода.
- **Незнакомый шрифт** (после обновления игры) → нейтральный fallback + запись в лог для добавления в `font_map`.
- **Изоляция сбоев** — хуки нашего плагина в try/catch: наша ошибка не роняет игру и не ломает AT.
- **Полный откат** — все правки в рантайме; удаление папки BepInEx возвращает игру в исходный вид.

### Проверка

1. **Smoke-тест запуска** — игра стартует, в логе BepInEx видно загрузку AT и нашего плагина.
2. **Тест шрифтов** — кириллица каждого стилевого шрифта рисуется в нужном стиле (скриншоты до/после).
3. **Тест вёрстки** — намеренно длинные русские строки не обрезаются.
4. **Тест перевода** — проход по меню/уровню: переведённое подменяется, непереведённое дампится.
5. **Тест отката** — удаление мода возвращает исходный вид.

Основной критерий — визуальный, поэтому ключевая проверка — **скриншоты в реальной игре**, а не только автотесты.

## 8. Вне области (YAGNI)

- Никакого машинного перевода в рантайме (онлайн-перевод выключен полностью).
- Не редактируем и не перепаковываем оригинальные ассеты игры.
- Не делаем поддержку других языков, кроме русского.
- Не строим собственный движок перехвата текста (используем готовый XUnity.AutoTranslator).
- Предпочитаем создание TMP FontAsset в рантайме (без Unity Editor); AssetBundle через Editor — только запасной путь, если рантайм-метод не даст приемлемого результата (см. §4).
- Дорисовка кириллических глифов вручную — только для уникальных ключевых шрифтов без аналога, не для всех.

## 9. Порядок реализации (этапы)

Выбран режим **ранней проверки**: техника готовится первой, но шрифты/вёрстку валидируем на реальной кириллице до того, как вкладываемся в полный перевод.

1. **Технический каркас мода** — BepInEx + XUnity.AutoTranslator (офлайн) + плагин `GunsawRusFonts` (загрузка кириллических fallback-шрифтов, подключение к FontAssets, базовая адаптивная вёрстка). Триаж шрифтов (§4).
2. **Ранняя проверка** — мини-перевод (меню, 10–20 строк), проверка шрифтов и вёрстки в реальной игре по скриншотам. Цель — поймать проблемы со стилем/обрезкой до полного перевода. При необходимости — правки шрифтов/вёрстки.
3. **Дамп строк** — проход по игре с рабочим модом + статическое извлечение из ассетов/DLL → полный список текста и реальный объём словаря.
4. **Контент** — глоссарий с нуля → черновой перевод по экранам → выверка заказчиком. Параллельно — финализация подбора декоративных кириллических шрифтов.
5. **Сборка и проверка** — упаковка в одну папку, тесты (§7), README.

## 10. Открытые вопросы к этапу реализации

- Точный объём текста (число строк) — измеряется первым дампом.
- Финальная таблица триажа шрифтов (какие имеют кириллицу / готовую версию / требуют подбора или дорисовки) — заполняется на первом шаге реализации.
- Конкретные кириллические шрифты-кандидаты под декоративные оригиналы — подбираются и согласуются.
